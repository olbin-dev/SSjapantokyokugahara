# リリースノート: Causa Core v1.0-alpha

本リリースは、Causa Core（Anchor DBエンジン）の最初のマイルストーンである v1.0-alpha の仕様および機能概要をまとめたものです。

## 1. バージョン v1.0-alpha に含まれるもの (Features)

- **データモデル定義**: `AnchorCase`, `UISnapshot`, `LLMProposal`, `HumanApproval`, `ExecutionEvent`, `OutcomeEvidence`, `ReplayPolicy`, `ReplayDecision`, `AnchorRecord`, `AuditSummary` のPythonデータクラス（Dataclasses）実装。
- **決定論的リプレイ判定機能**: 厳格な一致ルールとリスクしきい値に基づいて、自動実行の適格性を判定する純粋関数 `determine_replay_decision` および `determine_replay_decision_from_record` の実装。
- **監査ログ生成 (Audit Summary)**: 判定結果と理由コードを、人間が監査しやすい親切な自然文で説明・出力する `build_audit_summary` レイヤー。
- **抽象ストレージレイヤー**: データベースの実装に依存しない共通インターフェース `CausaRepository` の定義。
- **テストインフラ**: ローカルテスト用のインメモリDB（`InMemoryCausaRepository`）と、判定パイプライン全体をシミュレーションするための包括的で非実行な単体テスト群。

## 2. 本バージョンに明示的に含まれないもの (Non-Goals)

安全境界を維持するため、以下の項目は本リリースから厳格に除外されています：
- OS実行（実際のマウス・キーボード操作、コマンド実行など）
- 自律型リプレイ
- アクセシビリティAPIのクエリ
- Dify連携
- Sudoまたは特権昇格
- ネットワーク通信（外部ソケットやHTTP呼び出し）
- 破壊的操作
- 本番用データベース（Supabase、PostgreSQL、ローカル以外のSQLite）の使用

## 3. アーキテクチャフロー (Architecture Flow)

`AnchorRecord` ➡️ `InMemoryCausaRepository` ➡️ `determine_replay_decision_from_record` ➡️ `ReplayDecision` ➡️ `build_audit_summary` ➡️ `AuditSummary`

## 4. 安全境界に関する制約 (Safety Constraints)

`causa_core` ライブラリは厳格にロジックおよびデータ定義を行うモジュールです。意思決定を行い監査サマリーを生成しますが、ホスト状態を変更したり、特権を昇格させたり、外部ネットワークと通信する機能は物理的に持っていません。テストを含め、すべては実際のOS操作を誘発することなく決定論的に動作します。

## 5. ストレージ実装の方向性 (Storage Direction)

Causa Coreは明示的にストレージ非依存です。`InMemoryCausaRepository` はテストおよびローカルシミュレーション専用に用意されたものです。コアとなる判定ロジック（`decision.py`）は、いかなる特定のストレージ実装からも完全に切り離されています。

## 6. 既知の制限事項 (Known Limitations)

- `InMemoryCausaRepository` の `get_anchor_record` は現在、最初に一致した関連レコードを取得します。
- 一部のみが一致する「部分的UI一致」の場合、判定は厳格にリプレイを拒否し、LLMによる推論（`requires_llm_reasoning`）を要求します。ネイティブな曖昧（ファジー）マッチングはサポートされていません。

## 7. 次回のマイルストーン (Next Steps)

次のマイルストーンでは、コア決定ロジックの独立性を維持したまま、ローカルでのデータ永続化を行うための **SQLiteローカルアダプター** の開発を計画しています。
